# Phase 2.3: Webhook Trigger & Full Pipeline Execution — Design Spec

## Overview

n8n-orchestrated pipeline execution triggered via HTTP webhook. The API container exposes stateless `/execute/*` endpoints; n8n sequences them with error handling and status tracking.

## Architecture

```
POST /api/v1/pipeline/trigger
  -> Create pipeline_run (pending)
  -> POST n8n webhook (run_id, source_dir, tenant_id)
  -> Return run_id immediately

n8n workflow sequences:
  PATCH status=running
  POST /execute/bronze        -> loader.run()
  PATCH status=bronze_complete
  POST /execute/dbt-staging   -> subprocess: dbt run --select staging
  PATCH status=silver_complete
  POST /execute/dbt-marts     -> subprocess: dbt run --select marts
  PATCH status=gold_complete  -> success

  On error at any step:
  PATCH status=failed + error_message
```

## Models

```python
class TriggerRequest(BaseModel):
    source_dir: str = "/app/data/raw/sales"
    tenant_id: int = 1

class TriggerResponse(BaseModel):
    run_id: UUID
    status: str

class ExecuteRequest(BaseModel):
    run_id: UUID
    source_dir: str = "/app/data/raw/sales"
    tenant_id: int = 1

class ExecutionResult(BaseModel):
    success: bool
    rows_loaded: int | None = None
    error: str | None = None
    duration_seconds: float
```

## Key Decisions

1. **n8n orchestrates** (Option C) — leverages existing n8n infra, visual debugging, easy Phase 2.5-2.8 extension
2. **/execute/* endpoints are stateless** — executor does NOT update pipeline_runs; API layer handles status
3. **Bronze calls loader.run() directly** (same container) vs subprocess for dbt (no Python API)
4. **No new migrations** — pipeline_runs table from Phase 2.2 already has all needed columns
5. **Internal-only endpoints** — secured by Docker network boundary (port 8000 = 127.0.0.1)
